Lossless Compression: How Computers Store More with Less
Every time you send a file over the internet, store a photo, or zip a folder, a compression algorithm runs silently in the background. Lossless compression is one of the most important tools in computer science. It reduces the size of data without discarding a single bit of the original information. When you decompress the file, you get back exactly what you started with. No approximations. No shortcuts. The mathematics that make this possible are elegant, and understanding them gives you a clearer picture of how computers represent and store information.
This essay walks through the core ideas behind lossless compression: what redundancy means in data, how the major algorithm families exploit it, and why these techniques remain the backbone of modern data storage and transmission.
The Core Idea: Redundancy
Lossless compression works because real-world data contains redundancy. Redundancy means that some information appears more than necessary to convey the message. Natural language is a clear example. In English, the letter 'e' appears roughly 13% of the time, while 'z' appears less than 0.1%. If every character in a file uses the same number of bits regardless of frequency, you waste storage on rare characters and fail to take advantage of common ones.
Redundancy also appears in patterns. The string 'aaaaaaaaaa' contains ten identical characters. A smart system does not need to store each 'a' separately. It stores the instruction: repeat 'a' ten times. The more redundant your data, the more aggressively it compresses. A random file of noise compresses poorly because there is no pattern to exploit. A text document or source code file compresses well because language follows rules and patterns.
Claude Shannon formalized this idea in 1948 with information theory. His concept of entropy measures the average information content of a symbol in a message. A symbol that appears 50% of the time carries one bit of information. A symbol that appears 0.1% of the time carries about 10 bits. Shannon proved that no lossless compression system can compress data below its entropy. This gives us a hard theoretical floor and tells us exactly how much redundancy a dataset contains.
Entropy Coding: Matching Bits to Frequency
The oldest and most direct approach to lossless compression is entropy coding. The goal is simple: assign short codes to frequent symbols and long codes to rare ones. David Huffman invented his famous algorithm in 1952 as a graduate student at MIT. Huffman coding builds a binary tree from the bottom up. It starts by placing each symbol in a leaf node weighted by its frequency. It then repeatedly merges the two lowest-frequency nodes until one tree remains. The path from the root to each leaf becomes that symbol's code.
Consider a simple example. Suppose you have four symbols: A appears 50% of the time, B appears 25%, C appears 15%, and D appears 10%. Huffman coding assigns A the code '0' (1 bit), B the code '10' (2 bits), C the code '110' (3 bits), and D the code '111' (3 bits). A fixed-length code would use 2 bits per symbol. With Huffman coding, the average drops to 1.85 bits per symbol, a reduction of roughly 7.5%. The savings grow dramatically when frequency distributions are more skewed.
Arithmetic coding takes entropy coding further. Instead of assigning discrete codes to individual symbols, arithmetic coding represents an entire message as a single fractional number between 0 and 1. Each symbol narrows the interval based on its probability. The final number encodes the entire sequence. Arithmetic coding achieves compression ratios closer to the theoretical entropy limit than Huffman, especially for messages with symbols of unequal and non-power-of-two probabilities. It is computationally more expensive but produces tighter results.
Asymmetric Numeral Systems (ANS), developed by Jaroslaw Duda in 2009, represents the modern standard. ANS combines the compression efficiency of arithmetic coding with speeds that rival Huffman coding. Facebook's Zstandard and Apple's LZFSE use ANS. It has largely replaced both older methods in production systems where speed and compression ratio both matter.
Dictionary Coding: Replacing Patterns with References
Entropy coding exploits frequency. Dictionary coding exploits repetition. The idea is to build a dictionary of strings that appear multiple times in the data. When the encoder encounters a string already in the dictionary, it outputs a short reference to the dictionary entry instead of repeating the full string. The decoder uses the same dictionary to reconstruct the original data exactly.
Abraham Lempel and Jacob Ziv published two landmark algorithms in 1977 and 1978, known as LZ77 and LZ78. LZ77 uses a sliding window. As it reads through the input, it maintains a window of recently processed data. When it finds a match for the current input in that window, it outputs a back-reference: an offset indicating how far back the match starts, and a length indicating how many characters to copy. The compressor and decompressor maintain identical windows, so the decompressor follows the same references to reconstruct the data.
LZ78 builds an explicit dictionary. It starts with an empty dictionary and grows it by adding new strings as they appear. Terry Welch extended LZ78 in 1984 to create LZW, which became famous as the algorithm behind GIF images and the Unix compress utility. LZW initializes the dictionary with all single characters and adds longer strings dynamically as the encoder processes the input.
DEFLATE, designed by Phil Katz and used in ZIP, gzip, and PNG, combines LZ77 with Huffman coding. It first applies LZ77 to replace repeated strings with back-references. It then applies Huffman coding to the stream of literals and references. This two-stage pipeline exploits both types of redundancy: repetition and frequency skew. DEFLATE became one of the most widely deployed compression formats in history and remains the backbone of web content compression through the HTTP protocol.
Zstandard (zstd), released by Facebook in 2016, represents the modern evolution of this approach. It uses ANS entropy coding instead of Huffman and includes a training mode that builds a shared dictionary from a sample of representative data. When compressing many similar small files, a pre-trained dictionary eliminates the cold-start problem where early data has no dictionary entries to reference. Zstandard consistently achieves better compression ratios than gzip at significantly higher speeds.
Transform Coding: Restructuring Before Compressing
Some data becomes more compressible after a mathematical transformation that reorders its structure without losing any information. The Burrows-Wheeler Transform (BWT), published in 1994, rearranges the characters of a text by sorting all rotations of the string and taking the last column. The output is not shorter, but it groups similar characters together. Characters that frequently follow the same context end up adjacent. This makes the transformed data far more amenable to subsequent entropy coding.
BZip2 applies BWT followed by Move-To-Front coding and Huffman coding. MTF maintains a list of symbols ordered by recency. When it encodes a symbol, it outputs the symbol's position in the list and moves it to the front. Because BWT clusters identical characters, MTF produces many small numbers, which Huffman then encodes efficiently. BZip2 consistently outcompresses DEFLATE on text but runs slower and uses more memory.
Prediction by Partial Matching (PPM) builds a statistical model of the data as it processes it, predicting each symbol based on the preceding context of varying lengths. It uses higher-order contexts when available and falls back to shorter contexts when the longer ones have no data. The predictions drive arithmetic coding. PPM variants achieve some of the best compression ratios on text among general-purpose algorithms, though they run slowly and require significant memory.
Run-Length Encoding and Specialized Formats
Run-Length Encoding (RLE) is the simplest form of lossless compression. It replaces consecutive identical values with a count and the value. Instead of storing 'AAAAAABBB', RLE stores '6A3B'. RLE works poorly on most text files but excels on data with long runs of identical values. Fax machines use RLE because scanned documents consist mostly of white pixels with occasional black lines.
PNG images use a set of filters on each row of pixels before applying DEFLATE. The filters compute differences between adjacent pixels rather than storing raw values. Because adjacent pixels in natural images are usually similar, the differences cluster near zero, creating a highly skewed distribution that DEFLATE compresses well. FLAC audio compression uses linear prediction to model each audio sample from its predecessors and then losslessly encodes only the residual error. The residuals carry far less information than the raw samples.
The Fundamental Limits
Shannon's source coding theorem gives lossless compression its hard boundary. No algorithm can losslessly compress all inputs. This follows from a simple counting argument: there are more possible inputs than possible outputs of any shorter length. Some inputs must therefore either stay the same size or grow. Truly random data has maximum entropy and cannot compress at all. Encrypted data is indistinguishable from random and resists compression for the same reason. This is why you compress files before encrypting them, not after.
The gap between theoretical limits and practical algorithms has narrowed over decades. Modern compressors like LZMA and Zstandard operate within a few percent of Shannon entropy on typical text and code. Machine learning has begun to close this gap further. Large language models are extraordinarily accurate predictors of text. A compressor that uses an LLM to assign probabilities to each token and then applies arithmetic coding achieves compression ratios far beyond classical methods on natural language. The cost is speed: compressing with an LLM takes orders of magnitude longer than running gzip.
Why It Matters
Lossless compression sits inside every major file format, network protocol, and storage system you use daily. HTTP uses DEFLATE or Brotli to compress web pages in transit. Git uses zlib to store repository objects. PNG uses DEFLATE to store images. The algorithms described here collectively save petabytes of storage and bandwidth every day.
Understanding lossless compression teaches you something deeper about information itself. The more structured and predictable your data, the smaller it gets. A good compressor is, in a precise sense, a good predictor. The better it predicts the next symbol, the shorter its code for that symbol. Compression and understanding are the same operation.
From Huffman's binary tree to neural network predictors, the core goal has never changed: find the pattern, model it precisely, and store only the surprise.
